### ❌ Problèmes détectés (par gravité)

**T1 — CRITIQUE : le corps des pages est vide sans JavaScript.**
Test en production : un fetch de `https://thecosmo.app/` et `/guide` sans exécution JS retourne **uniquement le `<title>`**. Le contenu de secours est dans `<noscript>`, que la plupart des extracteurs (LLM, convertisseurs markdown, certains crawlers) **ignorent totalement**. Googlebot rend le JS (avec délai et budget), mais : Bing rend le JS de façon inégale, les crawlers IA (GPTBot, ClaudeBot, PerplexityBot) ne le rendent **pas du tout**.
→ *Fix* : étendre `prerender.mjs` pour injecter le contenu **en HTML visible** dans `<div id="root">` (ou juste avant), que React remplace à l'hydratation. Zéro navigateur requis au build, compatible avec l'approche actuelle.

**T2 — MAJEUR : soft-404 généralisé.** La route `*` fait `Navigate to="/"` : toute URL inexistante renvoie **HTTP 200 avec le contenu de la home**. Google classe ça en soft-404 et cela gaspille le budget de crawl.
→ *Fix minimal* : une vraie page 404 (composant NotFoundPage avec meta `noindex`) ; idéalement, servir un statut 404 pour les chemins hors liste via la config Vercel.

**T3 — MAJEUR : CSP bloque AdSense.** `script-src 'self' https://js.stripe.com` ne contient pas `pagead2.googlesyndication.com` : le script AdSense chargé dans index.html est **bloqué par la CSP** (erreur console + requête morte sur chaque page vue). Soit retirer la balise, soit compléter la CSP (`script-src`, `frame-src`, `connect-src` AdSense).

**T4 — MOYEN : URLs à tokens crawlables.** `/invite/:token` et `/org-invite/:token` ne sont ni dans robots.txt ni en noindex. Si un lien d'invitation fuite (mail forwardé, Slack public), Google peut indexer des URLs à token.
→ *Fix* : `Disallow: /invite/` + `Disallow: /org-invite/` dans robots.txt + meta noindex sur ces pages.

**T5 — MOYEN : Google Fonts render-blocking.** `<link href="https://fonts.googleapis.com/...">` = 2 connexions externes sur le chemin critique + question RGPD (transfert IP vers Google, jurisprudence allemande).
→ *Fix* : self-host Inter (fichiers woff2 dans `/public/fonts`, `font-display: swap`). Gain LCP estimé 100–300 ms.

**T6 — MINEUR :**
- `dateModified: 2026-05-31` codé en dur dans 3 blocs JSON-LD + sitemap → figer une date stale nuit à la fraîcheur perçue. Générer au build (`new Date()`).
- `contactPoint.email: contact@cosmo.app` alors que le domaine est **thecosmo.app** — si cosmo.app ne vous appartient pas, les emails partent chez un tiers. À corriger dans le JSON-LD Organization.
- `sameAs: []` vide — ajouter les profils sociaux dès qu'ils existent (X, LinkedIn, Product Hunt) : signal d'entité pour le Knowledge Graph.
- Meta `keywords` : ignorée par Google depuis 2009, inoffensive mais inutile.
- Title homepage = 78 caractères → tronqué en SERP (~60 max affichés).

### Core Web Vitals ⚠️ (non mesurés — à valider via PageSpeed Insights)

Risques identifiés dans le code : hero landing animé GSAP + SplitText (risque LCP si le H1 est masqué avant reveal — vérifier que le texte est peint avant l'animation), fonts externes (T5), chunk React + Router + Query au premier paint. Points positifs : chunks manuels, lazy routes, pas d'images lourdes au-dessus de la ligne de flottaison. **Hypothèse : LCP mobile 2,5–3,5 s** — mesurer avant d'optimiser.

### Budget de crawl / architecture

Non-problème aujourd'hui : 7 URLs. L'architecture est plate (profondeur 1), le maillage interne existe (footer + liens croisés guide/signup/home). Le sujet deviendra réel quand le blog existera : prévoir `/blog/<slug>` plat, fil d'Ariane (déjà schématisé), et sitemap généré au build.

## 2. Analyse On-Page — 68/100

### Page par page

| Page | Title (longueur) | Verdict |
|---|---|---|
| `/` | 78 car. ❌ trop long | Réduire à ~58 : « Cosmo — Tâches, habitudes et OKR dans une seule app gratuite » |
| `/guide` | 57 car. ✅ | Bon. Enrichir la description avec un bénéfice chiffré |
| `/signup` | 60 car. ✅ | OK |
| `/login` | 48 car. ✅ | OK (page à faible enjeu SEO) |
| `/cgu`, `/mentions-legales`, `/politique-confidentialite` | ✅ | OK — pages de conformité, enjeu nul |

**Meta description home** : 199 caractères → tronquée (~155 affichés). Garder l'accroche « Essayez sans inscription » dans les 150 premiers caractères — c'est votre meilleur différenciateur de CTR.

**Structure Hn** : la landing a un vrai `<h1>` + `<h2>` de sections ✅. Le **GuidePage** utilise des composants `SectionHeader` / `Step` — vérifier qu'ils rendent de vrais `<h2>/<h3>` sémantiques et non des `<p>` stylés (grep : aucun `<h1-h3>` natif trouvé dans GuidePage.tsx). Le guide est riche (515 lignes, 6 sections) mais ce contenu est invisible sans JS (cf. T1).

**Intention de recherche** : la home cible bien le transactionnel/navigationnel (« app productivité gratuite », « essayer sans inscription »). Il manque tout le **spectre informationnel** (voir §3) — c'est là que sont 90 % des volumes.

**E-E-A-T** : faible. Pas d'auteur identifié, pas de page « À propos », `sameAs` vide, aucune preuve sociale (avis, compteur d'utilisateurs, témoignages). Pour un SaaS productivité c'est secondaire (pas YMYL), mais une page À propos + des témoignages réels amélioreraient conversion ET confiance.

**FAQ** : 10 questions avec FAQPage schema — excellent travail, déjà au niveau des bonnes pratiques 2026. Les réponses citent les concurrents (Notion, Todoist) : bon pour le SEO sémantique.

## 8. SEO international

Aujourd'hui : mono-langue FR, hreflang correct. **Opportunité majeure à 6–12 mois** : le marché EN est ~15× le FR sur ces requêtes. Prérequis avant d'y penser : (1) avoir prouvé le playbook contenu en FR, (2) i18next (le CLAUDE.md l'anticipe déjà), (3) structure `/en/` en sous-répertoire (pas de ccTLD ni sous-domaine), hreflang bidirectionnel complet. Ne pas traduire automatiquement : adapter (les requêtes EN sont différentes, ex. « habit tracker app » ≫ « tracker d'habitudes »).

## 9. UX & SEO

Points forts : mobile-first réel (bottom nav, gestes, safe-area iOS), mode démo sans inscription = **friction d'essai la plus basse du marché** (excellent pour le taux de retour SERP→site, signal comportemental positif), dark/light, a11y suivie (docs + scans axe-core).
Points de friction : le CTA « Essayer la démo » doit être visible **au-dessus de la ligne de flottaison mobile** ; vérifier que les animations GSAP du hero ne retardent pas l'affichage du H1 (LCP + utilisateur `prefers-reduced-motion` déjà géré ✅) ; ajouter des breadcrumbs visibles quand le blog existera.

## 10. Optimisation IA / LLM — 25/100

**Test empirique réalisé pendant cet audit** : un agent IA qui fetch thecosmo.app ne récupère **que le `<title>`**. Concrètement, quand un utilisateur demande à ChatGPT/Claude/Perplexity « quelle app gratuite pour suivre tâches + habitudes + OKR ? », Cosmo est **incitable uniquement via les données d'entraînement, pas via le crawl temps réel**.

| Critère | État |
|---|---|
| Contenu lisible sans JS | ❌ Critique (T1) |
| Données structurées | ✅ Excellentes |
| FAQ format Q/R | ✅ 10 questions bien formulées |
| Pages piliers citables | ❌ Inexistantes |
| llms.txt | ❌ Absent (standard émergent, 30 min de travail) |
| Crawlers IA autorisés | ✅ robots.txt ne les bloque pas |

**Actions** : régler T1 (le HTML statique visible sert Google ET les LLM), publier `/llms.txt` (résumé du produit + liens vers les pages clés), garder les formulations « X est la seule application qui… » — les LLM citent volontiers les phrases définitionnelles nettes. Les futurs comparatifs « Cosmo vs X » sont le format le plus repris par Perplexity.

## 12. Roadmap

### 🔴 Priorité 1 — 0 à 7 jours (quick wins techniques)
1. Étendre `prerender.mjs` : contenu HTML **visible** (pas seulement noscript) pour `/`, `/guide` et futures pages — règle T1, le fix le plus rentable de tout l'audit.
2. Raccourcir title (≤ 60 car.) et meta description (≤ 155 car.) de la home.
3. `Disallow: /invite/` + `/org-invite/` dans robots.txt + noindex sur ces pages.
4. Corriger ou retirer AdSense de la CSP ; corriger l'email `contact@cosmo.app` → domaine réel ; `dateModified` dynamique au build.
5. Créer une vraie page 404 avec noindex.
6. **Ouvrir Google Search Console + soumettre le sitemap** (si pas déjà fait) — sans GSC, tout le reste est du pilotage à l'aveugle.
7. Publier `/llms.txt`.
8. Inscription sur 10 annuaires SaaS (Product Hunt, AlternativeTo, Tool Advisor, appvizer…).

### 🟠 Priorité 2 — 1 mois
9. Self-host des fonts Inter ; audit PageSpeed mobile et correction LCP si > 2,5 s.
10. Vérifier/corriger la sémantique Hn du GuidePage (vrais `<h2>/<h3>`).
11. Mettre en place la structure blog (`/blog/<slug>`, sitemap généré au build, Article schema, breadcrumbs).
12. Publier les 2 premiers articles (pilier OKR + comparatif Todoist).
13. Page « À propos » + remplir `sameAs` + témoignages réels sur la landing.

### 🟡 Priorité 3 — 3 mois
14. 2 articles/mois (calendrier §5) ; template OKR téléchargeable comme linkable asset.
15. Campagne d'outreach : articles « alternatives à X » existants + 1 article invité/mois.
16. Pages use-case (/pour-freelances, /pour-etudiants, /pour-managers) ciblant les requêtes commerciales.

### 🟢 Priorité 4 — 6 à 12 mois

19. Data study annuelle (données anonymisées agrégées) pour la presse.
